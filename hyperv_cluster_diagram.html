⚡ Live Migration — Zero Downtime
1
Admin initiates move — Right-click VM in Failover Cluster Manager → Move → Live Migration → HV-NODE2
2
Memory copy begins — NODE1 copies VM RAM pages to NODE2 over the 192.168.20.x Live Migration network
3
Storage stays put — VM disk files remain on C:\ClusterStorage (iSCSI). Both nodes already have access so no data moves
4
Final sync — Last few changed memory pages sent, NODE2 takes ownership in milliseconds
✓
Zero downtime — VM never stopped. Equivalent to VMware vMotion. Owner Node changes from NODE1 → NODE2
🔴 Automatic Failover — HA in Action
1
NODE1 fails — Power loss, OS crash, or cluster service stops. NODE1 goes silent on all networks
2
Heartbeat lost — NODE2 stops receiving cluster heartbeat from NODE1 on the management network
3
Quorum votes — NODE2 (1 vote) + Disk Witness (1 vote) = 2 votes. Majority reached. NODE2 declared authoritative
4
Storage claimed — NODE2 takes exclusive ownership of the cluster disks via iSCSI. Starts TEST-VM01 from shared storage
✓
VM back online — TEST-VM01 running on NODE2 in 30-90 seconds. Equivalent to VMware vSphere HA

✅ NORMAL OPERATION
NODE1 (1) + NODE2 (1) + Witness (1) = 3 votes total. Both nodes active. Cluster healthy. Any node can lose connectivity and the other two maintain quorum with 2 votes.
❌ NODE1 FAILS
NODE1 loses all votes. NODE2 (1) + Witness (1) = 2 votes. Still a majority out of 3. Cluster continues on NODE2 alone. No split-brain. VM restarts automatically.
⚠️ WITHOUT WITNESS
With only 2 nodes and no witness, if network breaks between them both see 1 vote. Neither has majority. Both shut down to prevent data corruption. This is split-brain protection — but it means total downtime.
💡 WHY A DISK WITNESS
The disk witness is on the iSCSI storage which both nodes can always reach independently. Even if the management network between nodes breaks, both nodes can still reach the witness disk to vote. Reliable tiebreaker.
